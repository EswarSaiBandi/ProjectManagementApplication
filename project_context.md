# Project Context: Project Studio (Construction Management SaaS)

## 1. Project Overview
We are building a production-grade construction project management platform.
**Key USP:** A "vernacular" interface where site workers report progress using voice (Hindi/Marathi/Tamil) which is automatically translated to English text.

The system consists of:
1.  **Web Dashboard (`apps/web`):** For Project Managers/Admins to schedule tasks, approve material requests, view financial health, and record audio updates.
2.  **Mobile App (`apps/mobile`):** For Site Supervisors to track attendance (GPS), record daily progress (Audio/Photos), and manage inventory (GRN).
3.  **Backend Services (`packages/backend`):** A lightweight Node.js server specifically for handling Google Speech-to-Text API and complex business logic not suitable for Edge Functions.
4.  **Supabase:** The primary backend handling Database, Authentication, and File Storage.

## 2. Tech Stack & Architecture

### **Architecture: Monorepo**
* **Manager:** `npm workspaces` or `TurboRepo`.
* **Structure:**
    * `apps/web` (Next.js)
    * `apps/mobile` (Flutter)
    * `packages/backend` (Express.js)
    * `supabase` (Config & Migrations)

### **Frontend: Web Dashboard**
* **Framework:** Next.js 14 (App Router).
* **Styling:** Tailwind CSS + ShadCN UI.
* **State Management:** React Query (TanStack Query) - *Best for Supabase data fetching.*
* **Audio:** Browser MediaRecorder API.

### **Frontend: Mobile App**
* **Framework:** Flutter (Dart).
* **Key Packages:**
    * `supabase_flutter`: For Auth & Database.
    * `geolocator`: For GPS Attendance.
    * `image_picker`: For Site Photos.
    * `record`: For Audio recording.

### **Backend Infrastructure**
* **Primary Database:** Supabase PostgreSQL.
* **Auth:** Supabase Auth (Email/Password, Phone OTP).
* **Storage:** Supabase Storage Buckets.
* **AI Service:** Google Cloud Speech-to-Text API (via `packages/backend`).

4. Database Schema (PostgreSQL / Supabase)
A. Core & Auth
Note: Users are managed via auth.users in Supabase. We use a public profiles table to store extra roles.

SQL

CREATE TABLE public.profiles (
    user_id UUID REFERENCES auth.users(id) PRIMARY KEY,
    full_name TEXT,
    role TEXT CHECK (role IN ('Admin', 'ProjectManager', 'SiteSupervisor', 'Client', 'Vendor')),
    phone TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE public.projects (
    project_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    project_name TEXT NOT NULL,
    client_id UUID REFERENCES public.profiles(user_id),
    start_date DATE,
    status TEXT CHECK (status IN ('Planning', 'Execution', 'Handover', 'Completed')),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
B. Planning & Execution (The Site Loop)
SQL

CREATE TABLE public.activities (
    activity_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    project_id BIGINT REFERENCES public.projects(project_id),
    name TEXT NOT NULL,
    start_date DATE,
    internal_duration_days INT,
    client_buffer_days INT,
    completion_percentage INT DEFAULT 0,
    status TEXT DEFAULT 'Not Started'
);

CREATE TABLE public.site_updates (
    update_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    activity_id BIGINT REFERENCES public.activities(activity_id),
    reported_by UUID REFERENCES public.profiles(user_id),
    
    -- Evidence
    audio_url TEXT, -- Link to Supabase Storage bucket 'audio-logs'
    transcript_text TEXT,
    remarks TEXT,
    
    progress_claimed_pct INT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
C. Resource Management
SQL

CREATE TABLE public.material_master (
    material_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    item_name TEXT NOT NULL,
    unit TEXT
);

CREATE TABLE public.purchase_requests (
    pr_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    project_id BIGINT REFERENCES public.projects(project_id),
    requester_id UUID REFERENCES public.profiles(user_id),
    status TEXT CHECK (status IN ('Pending', 'Approved', 'Rejected')),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE public.pr_items (
    item_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    pr_id BIGINT REFERENCES public.purchase_requests(pr_id),
    material_id BIGINT REFERENCES public.material_master(material_id),
    requested_qty DECIMAL,
    approved_qty DECIMAL -- Edited by PM
);

CREATE TABLE public.goods_received (
    grn_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    pr_id BIGINT REFERENCES public.purchase_requests(pr_id),
    received_by UUID REFERENCES public.profiles(user_id),
    received_date TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
D. Financials
SQL

CREATE TABLE public.transactions (
    transaction_id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    project_id BIGINT REFERENCES public.projects(project_id),
    type TEXT CHECK (type IN ('Credit', 'Debit')),
    category TEXT CHECK (category IN ('ClientPayment', 'VendorPayment', 'LabourPayout')),
    amount DECIMAL(15,2),
    payment_channel TEXT, -- UPI, NEFT, Cash
    receipt_url TEXT, -- Link to Supabase Storage bucket 'documents'
    transaction_date DATE
);
5. Storage Buckets (Supabase Storage)
We require the following public buckets:

site-evidence: Stores photos/videos uploaded by the Mobile App during Site Updates.

audio-logs: Stores the raw audio files (.m4a or .wav) recorded by Site Supervisors.

documents: Stores PDFs (Invoices, PRs, Handover Checklists).

6. Key Business Logic
Voice-to-Text Workflow
Mobile/Web: Records audio -> Uploads file to Supabase Storage (audio-logs).

Client: Sends the public_url of the audio to packages/backend API endpoint /transcribe.

Backend: Downloads audio -> Sends to Google Speech API -> Returns text.

Client: Displays text in "Remarks" field -> User edits -> User submits to site_updates table.
